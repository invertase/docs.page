The `@docs.page/cli` package is the local companion to [docs.page](https://docs.page) hosting. Run it from your project root to create starter files, see edits in the browser before you push, catch broken links and missing assets, and provision the in-docs AI chat agent for a GitHub repository.

## Overview

docs.page serves documentation from a public GitHub repository — there is no separate deploy dashboard. The CLI fills the gaps on your machine: scaffolding, checking, and live preview while you write.

| Command | Role in your workflow |
| --- | --- |
| `docs init` | Create `docs.json` and optional starter pages in a new or existing repo |
| `docs preview` | Watch local files and render changes in the docs.page preview UI |
| `docs check` | Lint configuration, links, and assets — a quality gate before you push |
| `docs agent` | Create or delete the in-docs chat agent for a repository |

## Quick start

```bash
npx @docs.page/cli init
npx @docs.page/cli preview
npx @docs.page/cli check
```

[Install the CLI](/concepts/cli/install) globally when you use these commands every day. For flag-level detail and exit codes, see [CLI reference](/reference/cli-reference).

## How it works

### `docs init`

`init` scaffolds the files docs.page expects. See [Initialize a project](/concepts/cli/init) for flags, overwrite behavior, and scripting in CI.

### `docs preview`

`preview` starts a local WebSocket server, renders your pages with the production pipeline, and streams results to `docs.page/preview`. See [Preview locally](/concepts/cli/preview) for ports, browser options, and limitations.

### `docs check`

`check` scans your project for broken internal links, missing assets, and unreachable external URLs. Each category uses severity `off`, `warn`, or `error`. The command exits with code `1` when any `error`-level issue is found. See [Check documentation](/concepts/cli/check) for CI setup and severity flags.

### `docs agent`

The `agent` command registers or removes in-docs chat credentials for a GitHub repository after verifying admin access. See [Manage the AI agent](/concepts/cli/agent) for create, delete, and `docs.json` configuration.

### Global options

Agent commands call the docs.page API. Set the API base URL with a global flag or environment variable:

```bash
docs --api-url https://your-api.example.com agent create --repo org/repo
```

```bash
export DOCS_PAGE_API_BASE=https://your-api.example.com
docs agent create --repo org/repo
```

When neither is set, the CLI uses the production docs.page API. `init`, `preview`, and `check` do not need an API URL.

## Design tradeoffs

**CLI over a web editor.** docs.page is Git-native: content lives in your repository and publishes on push. A terminal tool fits that model — scaffold into an existing repo, check in CI, and preview without committing work-in-progress branches.

**Hosted preview UI.** Local preview does not embed a full static copy of the docs.page app. Instead, your machine renders page content and the browser loads the production preview shell. You see real theme and layout behavior while file I/O stays on localhost.

**Check as a gate, not a deploy step.** `check` improves quality before push; it does not upload or publish anything. Hosting remains push-to-publish from GitHub.
