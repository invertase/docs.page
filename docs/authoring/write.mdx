**Before you begin**

- A [docs.page](https://docs.page) project with `docs.json` at the repository root and page files under `docs/`
- Basic Markdown familiarity: headings, lists, links, and fenced code blocks

## Write a new page

Each page is a `.mdx` or `.md` file under `docs/`. Work through these steps in order:

1. **Create the file** — add `docs/{path}.mdx` (or `.md`). The path without the extension becomes the URL.
2. **Set frontmatter** — declare `title` and `description` at the top of the file. See [Frontmatter](#frontmatter).
3. **Draft the body** — structure with `##` headings and write scannable prose. See [Write scannable prose](#write-scannable-prose).
4. **Add links, code, and media** — use root-relative links, fenced code blocks, and images where they help the reader. See [Links and assets](#links-and-assets).
5. **Publish the page to readers** — add it to the sidebar in [Organise](/authoring/organise), then confirm in [Preview](/authoring/preview).

## Page structure and prose

### Write scannable prose

Structure the body with `##` headings. The table of contents derives from heading levels — keep one main idea per section and nest subsections one level deeper.

Write so a busy reader can skim:

- **State the outcome first** — tell readers what they will accomplish before steps or settings
- **Keep paragraphs short** — split long blocks into bullets or numbered lists
- **Use imperative voice** — **Run**, **Set**, **Open**; not passive hedging ("It is recommended that…")
- **Format for clarity** — bold UI labels; put commands, paths, and keys in `backticks`

Match page shape to what the reader needs:

| Reader goal | Page shape | Link out for |
| --- | --- | --- |
| Learn by doing | Step-by-step walkthrough | Field lookup, API tables |
| Complete one task | Focused how-to with a verify step | Architecture essays |
| Look up a value | Reference tables or property rows | Tutorial narrative |
| Understand why | Explanation with a mental model | Numbered setup steps |

Stay in one mode per page. Link to Reference and Components instead of mixing exhaustive lists into prose.

**Before:** In order to preview locally, the user should run the docs.page CLI preview command from the project root.

**After:** Run `npx @docs.page/cli preview` from the directory that contains `docs.json`.

### Frontmatter

Start each page with YAML frontmatter between `---` delimiters. At minimum, set `title` and `description`:

```yaml
---
title: Install the CLI
description: Install the docs.page CLI on macOS, Linux, or Windows.
---
```

- **`title`**: page name in the browser tab, search results, and (when enabled) the on-page heading
- **`description`**: short summary for SEO and social previews. Write for readers: state the outcome or task, not a list of fields

Additional frontmatter keys control redirects, previous/next links, hero images, and indexing. See [Page frontmatter](/reference/page-frontmatter) for the full field list.

<Info>
  Global variables (`{{ dotted.path }}`) substitute in the MDX body only, not inside frontmatter. Define values in `docs.json` and reference them in prose or code fences. See [Global variables](/features/global-variables).
</Info>

### Where pages live

Store every page under `docs/`. Subdirectories are allowed. The file path (without the extension) becomes the URL path:

| File | URL |
| --- | --- |
| `docs/index.mdx` | `/` |
| `docs/getting-started.mdx` | `/getting-started` |
| `docs/api/authentication.mdx` | `/api/authentication` |

Create a new file when you add a topic. Rename the file and add a [redirect](/authoring/redirect-pages) when you move a published URL.

## Links and assets

### Internal links

Link to other pages with root-relative paths: no domain, no `.mdx` extension:

```mdx
See [Preview locally](/authoring/preview) before you open a pull request.
```

Root-relative links resolve on your live site, branch previews, and custom domains without hard-coding `https://docs.page/owner/repo`.

### Code examples

Wrap snippets in fenced code blocks and add a language tag so syntax highlighting and the copy button work:

````mdx
```bash
npx @docs.page/cli preview
```
````

Use `bash` for terminal commands, `json` for config, and `tsx` or `ts` for TypeScript. For multiple equivalent snippets (npm vs pnpm, curl vs fetch), use a [code group](/components/code-group) or [tabs](/components/tabs) instead of repeating blocks in prose. See [Code blocks](/components/code-blocks) for titles, diff markers, and other fence options.

### Images

Commit images under `docs/` (for example `docs/assets/dashboard.png`) and reference them with a leading slash:

```mdx
![Dashboard overview](/assets/dashboard.png)
```

Standard Markdown image syntax works. Switch to the [`<Image>`](/components/image) component when you need captions, light/dark variants, sizing, or per-image zoom control. Store screenshots as PNG or WebP; use SVG for diagrams and logos. Add meaningful `alt` text.

### Video and embeds

- **YouTube or Vimeo**: [`<YouTube>`](/components/youtube) or [`<Vimeo>`](/components/vimeo)
- **Self-hosted MP4/WebM**: [`<Video>`](/components/video) with a public `https` URL — repo-relative paths are not supported
- **Social posts**: [`<Tweet>`](/components/tweet)

## When to use components

Plain Markdown covers most writing. docs.page also ships **built-in MDX components** (tabs, callouts, steps, cards, and more) with no imports and no extra packages.

| Reader need | Start with |
| --- | --- |
| Body copy and lists | Plain Markdown |
| One highlighted snippet | Fenced code block |
| Platform or language variants | [`<Tabs>`](/components/tabs) or [`<CodeGroup>`](/components/code-group) |
| Prerequisites, cautions, tips | [`<Info>`](/components/callouts), [`<Warning>`](/components/callouts), or siblings |
| Numbered setup flow | [`<Steps>`](/components/steps) |
| API or config field rows | [`<Property>`](/components/property) on Reference pages |
| Section landing links | [`<Card>`](/components/card) and [`<CardGroup>`](/components/card) |

Author components with PascalCase JSX tags in your `.mdx` file. Unknown component names show an inline error in preview — match the tag to the [Components](/components) reference before you publish.

For props, child elements, and examples, use the [Components](/components) tab. For how the library fits the product, see [Features → Components](/features/components). Keep workflow pages focused on *when* to reach for a component; defer field-by-field lookup to the component reference.

## Verify

After you edit a page, use [Preview](/authoring/preview) to confirm layout and components, then run `docs check` to catch broken links and missing assets. When both look good, continue to [Publish](/authoring/publish).

## Troubleshooting

| Symptom | Likely cause | Fix |
| --- | --- | --- |
| Page returns 404 in preview | File is outside `docs/` or the path does not match the URL you opened | Keep pages under `docs/` and match the file path to the URL (no extension) |
| Internal link goes nowhere | Link includes a domain, file extension, or wrong path | Use root-relative paths like `/authoring/preview`. No `https://` and no `.mdx`. |
| Image does not render | Asset path is wrong or the file is not committed | Store images under `docs/` and reference them with a leading slash, e.g. `/assets/screenshot.png` |
| Inline component error in preview | Misspelled or unsupported component name | Match the tag to the [Components](/components) reference. Names are PascalCase. |
| `{{ variable }}` shows literally | Substitution used in frontmatter or the path is undefined | Use variables in the MDX body only; define values in `docs.json`. See [Global variables](/features/global-variables). |
