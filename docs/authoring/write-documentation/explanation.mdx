Use this layout when readers need to understand *why* something works the way it does — not just how to use it. Explanation pages build mental models and connect concepts.


<Info>
  Examples on the **Page templates** share a fictional **Coffee Shop Ordering API** so you can compare how the same product reads across documentation types.
</Info>


## Blueprint anatomy

Before writing, anchor yourself to the core constraints of the understanding-oriented quadrant:

- **The goal:** Build deeper conceptual understanding — background context, architecture, intent, and design decisions.
- **The tone:** Discursive, expository, and educational. Longer paragraphs are fine, but split walls of text and use headings generously.
- **The layout:** Overview → How it works → Related. Link out for procedures and field lookup. Weave rationale into `## How it works` when it helps — do not add a standalone tradeoffs section by default.

| Use explanation when… | Use something else when… |
| --- | --- |
| The reader asks *why* or *how does this fit together* | They need a guided first walkthrough → [Tutorial](/authoring/write-documentation/tutorial) |
| Architecture or state matters | They need steps for one task → [How-to](/authoring/write-documentation/how-to) |
| Code would distract from the concept | They need every field and error code → [Reference](/authoring/write-documentation/reference) |

## Example: The order fulfillment lifecycle

<Card>

Coffee Shop orders move through an asynchronous fulfillment pipeline. The API accepts payment and queues work immediately; the physical barista and espresso machine finish the drink on their own timeline.

### The concept

Pulling a shot of espresso takes a physical baseline of about 25 seconds. If order creation blocked until the cup hit the counter, HTTP clients would time out waiting for the barista. Async acceptance decouples **authorization** from **fulfillment**.

### How it works

When a client creates an order, the platform writes a ticket and returns `201 Created` with an `order_id`. Fulfillment proceeds in the background. Clients poll the order resource or subscribe to webhooks for status changes.

### The order state machine

Each order transitions through four explicit states:

1. **`received`** — Payment is authorized and the ticket is recorded.
2. **`queued`** — The order enters the barista's ticket queue.
3. **`brewing`** — The espresso machine is actively pulling the shot.
4. **`completed`** — The drink is on the pickup counter; a webhook notifies the client app.

Invalid transitions — for example, jumping from `received` to `completed` — are rejected server-side so clients never show a ready drink that is still being made.

### Design decisions

**Webhooks over long polling by default.** Most café apps care about the moment a drink is ready, not every intermediate state. Webhooks push `completed` once; polling remains available for clients that cannot receive callbacks.

**Explicit states instead of free-form strings.** Named states make integration tests predictable and give support staff a shared vocabulary when debugging stuck tickets.

</Card>
