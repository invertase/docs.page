title: Tutorial template
description: Structure guided walkthroughs with prerequisites, hands-on steps, and one clear thing to accomplish.
---

Use this layout when you want to guide someone through what they set out to do for the first time and help them start building a mental model — the moment the product clicks. The reader should finish with something working they can verify, not a menu of options or a deep dive into architecture.

<Info>
  **Tutorial vs how-to:** A tutorial is for readers who have never done this before — one guaranteed path through one thing they set out to do. Use a [how-to](/authoring/write-documentation/how-to) when they already know the basics and need to complete one specific task.
</Info>

These are **page templates** — layouts for individual `.mdx` files, not site themes, starter repos, or `docs init` scaffolds.


<Info>
  The examples on this page and the other **Page templates** use a fictional **Coffee Shop Ordering API**. The same product scenario appears in every template so you can compare how structure, tone, and code change by documentation type.
</Info>


## Blueprint anatomy

Before writing, anchor yourself to the core constraints of the learning-oriented quadrant:

- **The goal:** Provide a narrative, hands-on experience focused on one thing the reader sets out to do.
- **The tone:** Encouraging, authoritative, and instructional. Do not offer alternatives, deep explanations, or production hardening.
- **The layout:** Introduction → Prerequisites → Steps → Next steps. Use `<Steps>` for long procedures. Keep code blocks copy-paste ready.

| Use a tutorial when… | Use something else when… |
| --- | --- |
| The reader has never done this before | They need one specific task → [How-to](/authoring/write-documentation/how-to) |
| You want one guaranteed path through one clear objective | They need field-by-field lookup → [Reference](/authoring/write-documentation/reference) |
| Choices and tradeoffs would slow them down | They need background and *why* → [Explanation](/authoring/write-documentation/explanation) |

## Copy this template

Copy this raw skeleton into a new `.mdx` file to begin scaffolding your page.

````mdx title="docs/your-quickstart-page.mdx"
---
title: [Action] your first [Noun]
description: A short, 1-sentence summary of what the reader sets out to accomplish.
---

This page guides you through setting up your environment and [what they set out to do] in under 5 minutes.

## Prerequisites

- [Prerequisite 1, e.g., An API key]
- [Prerequisite 2, e.g., A specific dependency or tool]

## Steps

<Steps>
  <Step title="[Action 1: Setup]">
    Describe the first chronological step. Keep choices out of this flow.
  </Step>
  <Step title="[Action 2: Configuration]">
    Provide code snippets that are complete and copy-paste ready.
  </Step>
  <Step title="[Action 3: Execution]">
    Execute the command. Clearly state what the successful output looks like so they can confirm it worked.
  </Step>
</Steps>

## Next steps

- [Link to a logical next How-to Guide]
- [Link to the relevant technical Reference page]
````

## Example: Create your first coffee order

<Card>

### Prerequisites

- A Barista API key (use `test_key_demo` for sandbox requests)
- `curl` or any HTTP client

### Steps

1. **Get your Barista API key.** Copy your sandbox key from your project's developer console. For this walkthrough, you can use the mock key `test_key_demo`.

2. **Construct the payload.** Create a JSON object for a simple hot espresso:


   ```json
   {
     "item": "espresso",
     "size": "single"
   }
   ```


3. **Send the request.** Run this command in your terminal:


   ```bash
   curl -X POST https://api.coffeeshop.dev/v1/orders \
     -H "Authorization: Bearer test_key_demo" \
     -H "Content-Type: application/json" \
     -d '{"item": "espresso", "size": "single"}'
   ```


   You should receive a `201 Created` response with an `order_id`. Your virtual espresso is brewing.

### Next steps

- Customize an order with milk alternatives → see the [How-to template](/authoring/write-documentation/how-to)
- Look up every request field → see the [Reference template](/authoring/write-documentation/reference)

</Card>
