## File path to page URL

On the default [docs.page](https://docs.page) host, every page URL mirrors its path under `docs/`:

| File | Live URL |
| --- | --- |
| `docs/index.mdx` | `/` |
| `docs/next-steps.mdx` | `/next-steps` |
| `docs/usage.mdx` | `/usage` |
| `docs/guides/setup/index.mdx` | `/guides/setup` |

Nested directories become nested URL paths. A file named `index.mdx` inside a folder serves that folder's path.

## Production URLs

Production URLs serve content from the repository's **default branch** — usually `main`, but whatever GitHub reports as the default for that repo.

```
https://docs.page/{owner}/{repo}
https://docs.page/{owner}/{repo}/{page-path}
```

For example, `docs/usage.mdx` in `https://github.com/acme/handbook` is `https://docs.page/acme/handbook/usage`.

## Ref URLs

Append a `~{ref}` segment immediately after the repository name to serve a branch, commit, or pull request instead of the default branch:

```
https://docs.page/{owner}/{repo}~{ref}
https://docs.page/{owner}/{repo}~{ref}/{page-path}
```

| Ref segment | Resolves to |
| --- | --- |
| `~feature-docs` | Branch `feature-docs` |
| `~abc123…` (40-character SHA) | That commit |
| `~42` | Pull request #42 head branch |

Production URLs omit the `~ref` segment entirely.

## Alternate hosts

| Host type | URL pattern |
| --- | --- |
| Default | `https://docs.page/{owner}/{repo}[~{ref}]/{page-path}` |
| Vanity subdomain | `https://{owner}.docs.page/{repo}[~{ref}]/{page-path}` |
| Custom domain | `https://{domain}[~{ref}]/{page-path}` |

On vanity subdomains and custom domains, the owner and repository segments drop from the path. Ref segments work the same way — they appear immediately after the repository name (vanity) or at the start of the path (custom domain).
